# System Architecture - LifeMax

This document outlines the high-level architecture of LifeMax, a production-quality personal productivity operating system.

## Separation of Concerns

LifeMax is designed as a three-tier web application:
1. **Frontend (Client)**: A fast, desktop-first, responsive React web application.
2. **Backend (Server)**: A Node.js Express REST API, written in TypeScript.
3. **Database (Storage)**: A PostgreSQL database hosted on Neon, managed via Prisma ORM.

## Backend Layered Architecture

To prevent code duplication, enhance testability, and keep the application maintainable, the backend follows a strict layered architecture:

```
Request ➔ Routes ➔ Controllers ➔ Services ➔ Repositories ➔ Database (Prisma)
```

- **Routes**: Mapped directly to specific Controller handlers. No business logic exists here.
- **Controllers**: Responsible for receiving request data, validating payloads (using Zod), calling appropriate Services, and sending responses back to the client.
- **Services**: Contain the core business logic, calculations, workflow validations, and orchestration. They interact with Repositories to query/persist data.
- **Repositories**: Encapsulate database access via Prisma. If schema interactions change, only this layer is affected.
- **Prisma Client**: Autogenerated database access layer from the `schema.prisma` definitions.
